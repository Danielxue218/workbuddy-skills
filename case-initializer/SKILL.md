# 案件初始化 + 法律意见出具 Skill

> 角色：律师助理 → 策略合伙人（流程中自动切换）
> 风格：遵循 writing-style-profile 配置

---

## 一、触发条件

用户在对话中说以下任一内容时加载本 Skill：
- "初始化案件 [案件文件夹名]"
- "读取 [路径] 并出具法律分析意见"
- "分析这个案子 [路径]"
- "帮我看看 [案件名]"

---

## 二、工作流（7 步骤）

### Step 1: 确认案件来源路径

**输入约定**：用户提供案件文件夹路径（云端），或只提供案件名。

- 如用户给全路径 → 直接使用
- 如用户只给案件名 → 在 `D:\1.观韬云同步文件夹\1.诉讼案件\2026年观韬诉讼案件\` 下查找匹配文件夹
- 如用户只说"当前案件" → 检查 `D:\workbuddy\cases\` 下最近创建的文件夹

**确认后输出**：
```
已定位案件文件夹：[完整路径]
案件编号：[DJ/DX_YYYYMMDD_案件名]
开始扫描文件……
```

### Step 2: 扫描云端源文件夹

使用 `ls -R` 递归列出云端文件夹全部文件。

对扫描结果进行分类：
| 类别 | 匹配规则 | 目标子文件夹 |
|------|----------|-------------|
| 判决书/裁决书 | 文件名含"判决书""裁定书""裁决书" | 01-判决书\ |
| 证据材料 | 文件名含"证据""银行流水""合同""协议""凭证" | 02-证据材料\ |
| 法律文书 | 文件名含"起诉状""答辩状""代理词""申请书""备忘录" | 03-法律文书\ |
| 沟通记录 | 文件名含"微信""邮件""会议记录""沟通" | 04-沟通记录\ |
| 其他 | 无法分类 | 保留在根目录，标记待分类 |

**核心文件识别**（优先级）：
1. 判决书/裁决书 → 最高优先级（包含法院认定事实）
2. 起诉状/仲裁申请书 → 了解原始诉求
3. 证据目录 → 了解证据体系
4. 合同/协议 → 了解基础法律关系
5. 沟通记录 → 了解背景

### Step 3: 深度读取核心文件

按优先级依次读取（Read tool），提取关键信息：

| 文件类型 | 提取内容 |
|----------|----------|
| 判决书/裁决书 | 案号、法院、当事人、代理人、"经审理查明"事实段落、"本院认为"裁判理由、判决结果 |
| 起诉状/仲裁申请书 | 当事人信息、诉讼/仲裁请求、事实与理由概述 |
| 合同/协议 | 签约方、标的、关键条款（价款/期限/违约责任/争议解决） |
| 证据目录 | 证据编号、名称、类型、证明事项 |
| 沟通记录 | 关键对话节点（日期+内容摘要） |

**读取限制**：
- PDF 文件使用 Read tool（支持 PDF 解析）
- DOCX 文件尝试 Read tool（如不支持则标注"需手动提取"）
- 单个文件超过 50 页 → 优先读取前 10 页 + 关键章节

### Step 4: 初始化本地工作文件夹

在 `D:\workbuddy\cases\{案件名}\` 下创建标准 4 子文件夹结构：

```
D:\workbuddy\cases\{案件名}\
    ├── 01-判决书\
    ├── 02-证据材料\
    ├── 03-法律文书\
    ├── 04-沟通记录\
    └── case-memory.md        ← 案件事实卡片（Step 5 生成）
```

**命名规则**：
- 案件名沿用云端文件夹名（不含 DJ/DX 前缀和日期）
- 如云端为 `DJ_20260407_江海 Vs.中青_物业合同纠纷`，本地为 `江海Vs中青_物业合同纠纷`

**注意**：不复制文件内容到本地（节省空间），仅创建文件夹结构和案件卡片。

### Step 5: 生成案件事实卡片（case-memory.md）

基于 Step 3 提取的信息，按照 `templates/case-memory.md` 模板生成。

**卡片内容**：
1. 基本信息卡（案号/法院/当事人/案由/标的额/程序阶段/代理律师）
2. 事实时间轴（关键日期+事件，按时间排序）
3. 法律关系图（文字描述：当事人关系+争议标的+法律连接点）
4. 争议焦点归纳（原告主张+被告抗辩+法院认定/我方初步判断）
5. 证据目录索引（证据名称+证明事项+页码/来源）
6. 程序节点（立案日/开庭日/举证期限/判决日等关键程序日期）

**输出**：写入 `D:\workbuddy\cases\{案件名}\case-memory.md`

### Step 6: 出具《法律分析意见与工作思路》

**角色切换**：此时从"律师助理"切换为"策略合伙人"语气和思维模式。

严格按照 writing-style-profile 的 IRAC 五段式框架输出：

```
一、事实定性与法律关系识别
  - 案情简述：以极简语言还原客观事实，剔除情绪化描述
  - 法律关系定性：明确基础法律关系
    （如：名为股权转让实为让与担保、合作开发房地产纠纷等）

二、核心法律研判
  - 适用规则：援引《民法典》、相关司法解释及会议纪要的精准条文
  - 实务分析：结合当前司法辖区的裁判口径，分析权利义务边界
  - 优劣势评估：从证明责任分配角度，客观评估我方立场

三、争议焦点提炼 (Core Issues)
  - 1. [事实争点：如是否存在口头变更合同的行为]
  - 2. [法律争点：如违约金过高调整的裁量标准]
  - 3. [程序争点：如管辖异议、诉讼时效等]

四、诉讼/非诉应对策略
  - 预判结果：给出倾向性判断（不作绝对承诺）
  - 行动路线：包括但不限于 诉前保全/证据固化/发函施压/谈判方案
  - 方案对比：如有多方案，做对比表

五、补充提问与材料清单
  > **必要性说明**：分析现有信息对定性判断的影响
  - **补充问题**：[针对案情断裂处的针对性追问，基于"事实链条中断点"或"证据效力瑕疵点"]
  - **补充材料**：[具体到证据名称，如：原始载体、财务凭证、章程修正案等]
```

**约束条件**（强制）：
1. **法言法语强制性**：严禁口语化表达，使用规范法律术语
2. **裁判思维介入**：分析不仅限于条文，必须提及主流司法实务中的裁判倾向或类案处理逻辑
3. **补充清单关联性**：基于"事实链条中断点"或"证据效力瑕疵点"提出，严禁万金油清单
4. **免责声明**：结尾附 writing-style-profile 中的标准免责模板

**输出**：写入 `D:\workbuddy\cases\{案件名}\03-法律文书\法律分析意见与工作思路.md`

### Step 7: 总结与下一步建议

输出给用户的总结：

```markdown
## 案件初始化完成

**案件**：[案件名]
**本地工作区**：`D:\workbuddy\cases\{案件名}\`

### 已生成文件
1. `case-memory.md` — 案件事实卡片
2. `03-法律文书\法律分析意见与工作思路.md` — 初步法律分析意见

### 文件分类结果
| 类别 | 数量 |
|------|------|
| 判决书 | X 份 |
| 证据材料 | X 份 |
| 法律文书 | X 份 |
| 沟通记录 | X 份 |
| 待分类 | X 份 |

### 下一步建议
1. 审阅《法律分析意见》，补充或修正分析方向
2. 收集"补充材料清单"中的缺失材料
3. 如需发起诉讼/仲裁，说"起草起诉状/仲裁申请书"
4. 如需深度法律检索，说"检索类案 [关键词]"（需 MCP 北大法宝）
```

## 四、输入/输出规范

### 输入格式

```
用户："初始化案件 [案件文件夹名]"
   或 "读取 [完整路径] 并出具法律分析意见"
   或 "分析这个案子"（需配合当前上下文判断）

可选参数：
- 是否跳过文件分类 → 默认：否
- 是否同时激活 MCP 检索 → 默认：否（MCP 未激活时降级为提示用户手动检索）
```

### 输出格式

```
1. 案件卡片 → Markdown 文件（case-memory.md）
2. 法律分析意见 → Markdown 文件（03-法律文书\法律分析意见与工作思路.md）
3. 摘要信息 → 对话中直接输出
```

## 五、错误处理

| 错误场景 | 处理方式 |
|----------|----------|
| 云端路径不存在 | 提示用户确认路径，列出相似文件夹名供选择 |
| 云端文件夹为空 | 提示"文件夹为空"，询问是否有其他材料来源 |
| 核心文件无法读取（PDF加密等） | 标注"⚠️ 无法读取"，提示用户手动提供内容或解锁 |
| 本地路径已存在 | 询问"是否覆盖？"或自动追加日期后缀 |
| 文件分类无法判断 | 放入"其他"类别，提示用户手动分类 |
| MCP 未激活但需要检索 | 降级：输出"建议手动检索"提示 + 检索关键词推荐 |
